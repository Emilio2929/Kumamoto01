Plan de Desarrollo, Sprints y Backlog Detallado - Proyecto Kumamoto

Proyecto: Sistema de Gestión Escolar "Proyecto Kumamoto"

Duración Total: 6 Semanas (42 Días Calendario / 30 Días Hábiles)

Distribución: 3 Sprints de 2 semanas cada uno (14 días naturales por Sprint)

Metodología: Scrum / Agile

Stack Tecnológico Implementado: Backend (.NET 8/9 C#, Entity Framework Core), Frontend (Angular, SCSS, CSS Grid), Base de Datos (SQL Server / Oracle), DevOps (Docker, Docker Compose, Render).

Importante

Todas las tareas, módulos y artefactos descritos a continuación reflejan de forma exacta la estructura del código fuente implementado en el repositorio (controladores/endpoints mínimos en C#, servicios de dominio, componentes de Angular, Docker y manifiestos de nube en Render).

🏃‍♂️ Sprint 1: Arquitectura Base, Seguridad, Autenticación y Gestión Institucional

Período: Días 1 al 14 (Semanas 1 y 2)
Objetivo del Sprint: Establecer la base arquitectónica del sistema en el Backend (.NET) y Frontend (Angular), configurar la base de datos con Entity Framework Core, implementar el sistema de seguridad JWT y construir el portal de inicio de sesión junto con la gestión administrativa central (Usuarios, Año Académico, Grados, Secciones y Cursos).

Sprint Backlog Detallado - Sprint 1

ID Historia de Usuario / Módulo Tareas Técnicas (Implementación Real) Criterios de Aceptación
SB1-01 Configuración de Infraestructura y ORM
  • Configuración del proyecto ASP.NET Core Web API (Kumamoto.API.csproj).
  • Definición de cadenas de conexión y entornos en appsettings.json y appsettings.Development.json.
  • Modelado de entidades base en EF Core: Usuario.cs, Rol.cs, NivelEducativo.cs, PeriodoAcademico.cs, Grado.cs, Seccion.cs, Curso.cs.
  • Creación y ejecución de migraciones iniciales en la base de datos.
 La API compila correctamente, se conecta a la base de datos y genera las tablas relacionales sin errores.
SB1-02 Seguridad y Autenticación JWT
  • Desarrollo de TokenService.cs para la emisión y validación de tokens JWT con claims de roles (Directora, Docente, Auxiliar, Padre).
  • Desarrollo de AuthEndpoints.cs para manejar login, validación de credenciales y refresco de tokens.
  • Implementación de defensas HTTP (Rate Limiting, CSP, X-Frame-Options) en Program.cs.
  • Implementación de sanitización de entradas (trimming) para evitar fallos en contraseñas.
  • Desarrollo de lógica de recuperación de contraseña conectada a EmailService.cs.
 Los endpoints devuelven tokens JWT válidos según el rol. Las contraseñas con espacios en blanco accidentales se procesan correctamente.
SB1-03 Frontend: Portal de Login y Comunicados
  • Configuración de rutas principales en Angular (app.routes.ts) y estilos globales (app.scss).
  • Desarrollo del componente login.component.ts y su maquetación con CSS Grid en login.component.scss (diseño estático moderno y limpio sin carrusel).
  • Creación del modal de detalle de "Comunicados" en el login, asegurando escalado correcto de imágenes y scroll vertical completo sin recortes.
 El login es 100% responsive en móviles, tablets y escritorio. El modal de comunicados muestra contenido extenso e imágenes adaptadas perfectamente.
SB1-04 Gestión de Usuarios y Personal (Admin)
  • Desarrollo de endpoints CRUD en C#: UsuariosEndpoints.cs y AdministrativosEndpoints.cs.
  • Implementación de vistas en Angular para la Directora: módulos usuarios y personal dentro de features/dashboard/directora/pages.
  • Integración de servicios de frontend para listado, creación, edición y bloqueo de usuarios.
 La Directora puede administrar el personal institucional, asignar roles y gestionar accesos desde su panel.
SB1-05 Configuración de Año Académico y Estructura
  • Desarrollo de endpoints de gestión curricular: ConfiguracionAnioEndpoints.cs, GradosSeccionesEndpoints.cs, CursosEndpoints.cs.
  • Creación de las vistas de Angular para la Directora: configuracion-anio, grados-secciones, cursos, y estructura.
 Permite abrir/cerrar años académicos y estructurar los grados, secciones y cursos disponibles en la institución.

Línea de Tiempo Diaria - Sprint 3

Días Actividad Técnica Principal Entregables / Archivos Involucrados
Días 29-30 Modelado de entidades para el registro de incidencias conductuales y seguimiento de alumnos en riesgo. Incidencia.cs, AlumnoRiesgo.cs, AlertaRiesgo.cs, DesbloqueoCalificacion.cs.
Días 31-33 Desarrollo del servicio de alertas tempranas y optimización de consultas SQL con índices y LEFT JOINs. EarlyWarningService.cs, optimize_tutoria_queries.sql, optimize_attendance_reports.sql, DashboardEndpoints.cs.

SB3-01 Tutoría, Incidencias y Alerta Temprana de Riesgo
  • Modelado de entidades de seguimiento: Incidencia.cs, AlumnoRiesgo.cs, AlertaRiesgo.cs, Notificacion.cs.
  • Desarrollo de lógica de servicios de dominio: EarlyWarningService.cs, AlertaTempranaService.cs, y RiesgoService.cs.

 El sistema detecta y alerta automáticamente sobre alumnos con acumulación de inasistencias, bajo rendimiento o incidencias conductuales graves.
SB3-02 Flujo de Autorización de Desbloqueo de Notas
  • Modelado de la entidad DesbloqueoCalificacion.cs.
  • Desarrollo de DesbloqueoNotasEndpoints.cs garantizando la regla de negocio de "un solo uso" (el permiso se revoca automáticamente tras guardar la modificación de la nota).
  • Implementación de la vista desbloqueo-notas en el dashboard de la Directora (configurada como vista por defecto al iniciar sesión) con filtros de Bimestre y Semanas.
  • Resolución de advertencias de nulabilidad (CS8602) en consultas LINQ.
 La Directora puede otorgar permisos temporales de modificación de notas. Una vez que el docente realiza el cambio, el candado se cierra automáticamente.
SB3-03 Portal de Padres y Comunicación Institucional
  • Desarrollo de endpoints de consulta externa: PadresEndpoints.cs y ComunicadoEndpoints.cs.
  • Construcción del Portal de Padres en Angular (features/padres/pages): dashboard-padre, asistencia-diaria, horario, incidencias, y notas-padre.
  • Creación de la vista de gestión de comunicados (comunicados) en el portal administrativo de la Directora.
 Los padres acceden a un portal seguro para ver en tiempo real las notas, asistencia, horarios y comunicados oficiales de hijos.
SB3-04 Pruebas de Integración y Calidad (QA / UAT)
  • Ejecución de pruebas de integración en flujos críticos (Cálculo de promedios, Desbloqueo de notas y Alertas tempranas).
  • Verificación de seguridad en endpoints mediante simulación de peticiones concurrentes (Rate Limiting y validación de claims JWT).
  • Pruebas de Aceptación de Usuario (UAT) del diseño responsive y usabilidad en los portales de Directora, Docente, Auxiliar y Padres.
 Todos los módulos operan sin errores de servidor (500), los cálculos de notas son exactos y la Directora da su conformidad formal a los flujos.
SB3-05 Containerización Docker y Despliegue en Render
  • Creación de archivo de manifiesto para la plataforma en la nube Render (render.yaml).
  • Implementación de Dockerfiles multi-etapa optimizados para el Backend (.NET) y Frontend (Angular).
  • Configuración de orquestación local y de servicios mediante docker-compose.yml para enlazar la base de datos, backend y frontend en contenedores aislados.
 La aplicación completa (Backend, Frontend y Base de Datos) se levanta y ejecuta de forma estable en contenedores Docker mediante un solo comando (docker-compose up) y se despliega en Render.
